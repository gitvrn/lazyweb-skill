Design Research: Transactional Email API — Marketing Landing + Feature Page

Competitive design research for a code-first transactional email API for developers (a Resend / Postmark / Loops competitor). Scope: the marketing landing page and one key feature page. Desktop/web marketing pages only.

FOR THE CODING AGENT

Lead the hero with a real, runnable code block — the single send call — not a hero illustration; that is the one move every strong dev-API page in this corpus makes.

LAZYWEB REPORT — AGENT HANDOFF
Use this report as a starting point for implementing a transactional email API marketing landing + feature page using competitor-validated design patterns.

TOP RECOMMENDATIONS (do first):
1. Put a runnable send-email code block (curl + one SDK tab) in the first fold of the hero, beside the headline — make it the product visual, not a stock illustration.
2. Build the feature/"how it works" page as a 3-step scannable card grid (Send -> Track -> Deliver), each card pairing one capability with a small UI/code proof.
3. Add a developer social-proof strip directly under the hero: recognizable logos + a one-line trust metric (e.g. emails sent / uptime), not a testimonial wall.

INDEX ON: code-in-hero (5 of 9 landing refs); 3-up feature/how-it-works card grids (5 of 9); logos+metric trust strip placed high (4 of 9).
DO NOT OVER-INDEX ON: enterprise "Talk to sales / Request access" gated CTAs (sales-led, hurts self-serve dev signup); dark crypto/finance theming (off-audience); the in-corpus email-API hero screenshot whose brand is ambiguous (crawl-seed names conflict — verify before citing as a competitor).

Patterns

Recurring, table-stakes moves across the corpus. Each card shows the shared pattern with the real screenshots that exhibit it — not just a list of brand names.

Code / terminal in the hero, not an illustration

Strong seen in 5 of 9 landing refs Directional · no outcome data

Table stakes for dev tools: the first fold shows the product as runnable code or a terminal, so the visitor sees the integration in the first three seconds. For an email API the send call is literally the demo.

Feature / “how it works” as a scannable card grid

Strong seen in 5 of 9 refs Directional · no outcome data

Below the hero, capabilities are broken into a grid of cards (often 3-up), each with one short benefit + a small UI or code proof. This is the dominant structure for the feature page itself.

Developer social proof: logos + a hard number, placed high

Directional seen in 4 of 9 refs Directional · no outcome data

Trust is carried by a logo strip plus one quantified claim (teams worldwide, % improvement, scale), usually just under the hero — not by a long testimonial wall. Developers buy logos + numbers faster than quotes.

Anti-Patterns

What not to copy for a self-serve developer API. Each is shown with the real screenshot of the pattern.

Sales-gated primary CTA (“Request access” / “Talk to an expert”)

Avoid 2 of 9 refs lead with this Directional

Gating the hero behind sales contact reads as enterprise/B2B and kills a developer’s “try it in 5 minutes” expectation. A Resend/Postmark-class product should lead with “Start free” + the code, not a form.

Unique Angles

The “you have to see it” details worth stealing — each shown as the specific cropped screenshot of that exact element.

Code that returns a visible result

Directional 1 clear in-corpus example Single-source · off-category

Coiled doesn’t just show a snippet — it shows the snippet’s output (run status, memory). For an email API, render the 200 OK + message id (and an open/delivered event) right under the call so the hero proves delivery, not just syntax.

Live, in-page API playground as social proof

Directional 1 clear in-corpus example Single-source · adjacent-category

Photoroom embeds an actual API Playground module on the marketing page — visitors run the product before signing up. A “send a test email to yourself” widget would be the email-API equivalent and a strong differentiator.
