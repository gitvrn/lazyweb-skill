Design Research: Transactional Email API

Marketing landing page + a key feature page for a developer-first transactional email API (a Resend / Postmark / Loops competitor). Code-first hero · "how it works" / feature section · developer social proof. Desktop / web only.

FOR THE CODING AGENT

Put a runnable send-an-email code block in the hero — that single move is the strongest, most repeated pattern across every developer-API page in this corpus.

LAZYWEB REPORT — AGENT HANDOFF
Use this report as a starting point for implementing a transactional-email-API landing page + feature page using competitor-validated design patterns.

TOP RECOMMENDATIONS (do first):
1. Hero = headline + a real, copy-pasteable "send your first email" code/curl block (not an illustration); primary CTA "Get started", secondary "View docs".
2. Add a quantified deliverability/scale trust strip directly under the hero (e.g. "X emails/mo · 99%+ delivered") plus a row of real customer logos.
3. Feature page = a numbered "send your first email" how-it-works flow (Install SDK → Add key → Send → see it in the dashboard) shown with the actual dashboard UI.

INDEX ON: code-in-the-first-fold (seen in 6 of 11 references); quantified-scale + logo trust strip (5 of 11); a stepwise "first email" onboarding tied to a dashboard screenshot (3 of 11).
DO NOT OVER-INDEX ON: dark "terminal" theme as a requirement (aesthetic-only, 4/11, not tied to any outcome data); per-request pricing copy lifted from off-category APIs (single-source); any "outperforms competitor" deliverability claim — none of this is measured.

Anti-patterns

A lifestyle/marketing hero that hides the product

Avoidcontrast example, 1 of 11 Directional

For a developer audience, a polished marketing hero that leads with brand storytelling and dashboards-as-decoration — with no visible code — buries the one thing a dev came to evaluate. The shot below is a strong general landing page; the lesson is what's missing in the first fold for a code-first product, not that the page is bad.

Turning the feature page into a pricing wall

Avoid Single-source · off-category

Leading the feature page with tiered plan cards and "$ per million / talk to sales" pushes the developer toward a buying decision before they've sent a test email. Keep pricing on /pricing and link to it; the feature page's job is first success. The Apollo example is a pricing page (off-category for "feature page"), shown only to mark the trap.

Unique angles

Make the API runnable on the page itself

Directional2 standout references Directional

The most "you have to see it" detail: a live, in-page playground (or a literal curl you can copy) that lets a developer get a real response before creating an account. For an email API this maps to a "send a test email to yourself" widget in the hero.

Dark, terminal-forward developer aesthetic

Directionalseen in 4 of 11 references Directional · aesthetic-only

A recurring look (not a proven lever): dark backgrounds with terminal/code panels signal "built for engineers". Treat as optional styling — the code block earns trust, the dark theme just dresses it. No outcome data backs the theme itself.
