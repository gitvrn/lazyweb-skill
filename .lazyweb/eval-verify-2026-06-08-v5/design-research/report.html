Design Research: Transactional Email API — Marketing Landing + Feature Page

Competitive design research for a new developer-first transactional email API (a Resend / Postmark / Loops competitor). Scope: web/desktop marketing landing page + one key feature page. Goal: a code-first hero, a clear "how it works" / feature section, and credible developer social proof.

FOR THE CODING AGENT

Lead the landing page with a runnable code block in the hero — every strong reference in this corpus puts the product (code, command, or trace) in the first fold, not an illustration.

TOP RECOMMENDATIONS (do first):
1. Put a runnable code/command block in the hero (curl POST /emails + 3-line SDK send), paired with one primary CTA — not a stock illustration.
2. Build the feature page as a numbered "how it works" / quickstart: npm install -> init with API key -> send, each step beside its real code snippet.
3. Add a developer social-proof strip: customer logos + one quantified deliverability/volume stat (e.g. "99% deliverability", "N emails/mo"), placed directly under the hero.

INDEX ON: code/command-in-hero (8 of 11 dev-tool refs), numbered install->init->send quickstart with snippets (5 of 11 refs), logos+quantified-stat trust strip (5 of 11 refs).

Patterns

Code / command lives in the hero

Strong seen in 8 of 11 references Directional · visual prevalence

Table stakes: the best developer-API and dev-tool pages put a runnable code block, a one-line command, or a live terminal/trace in the first fold instead of a stock illustration.

"How it works" = numbered install → init → send quickstart

Strong seen in 5 of 11 references Directional · visual prevalence

Table stakes for the feature page: walk the developer from npm install to a successful call in three labeled steps, each with its real snippet — not prose.

Developer social proof = logos + one quantified stat

Directional seen in 5 of 11 references Directional · visual prevalence

Credibility for developers comes from a trusted-by logo row plus a single hard number (deliverability %, volume/mo, uptime) — placed right under the hero, not buried mid-page.

Usage-based pricing with a volume selector

Directional seen in 3 of 11 references Directional · visual prevalence

API products let buyers self-estimate cost: a "pay as you grow" tier ladder, often with a volume dropdown, plus a comparison table — reduces "contact sales" friction for developers.

Anti-Patterns

Borrowing the dark crypto/web3 aesthetic instead of the code

Avoid seen in 2 of 11 references Directional · off-audience

What not to copy: the heavy dark crypto-exchange styling. The transferable idea is the in-hero code block, not the theme — a generic transactional-email buyer is not a crypto-native and the styling can read as off-trust.

Gating the hero behind a "request access" signup form

Avoid no in-corpus screenshot captured Single-source / directional

What not to copy: making a contact/"request an invite" form the hero (seen on some SDK-beta pages in the wider corpus, e.g. tapmedia QR-SDK and Citymapper developer-access). For a self-serve email API, a developer should reach a runnable snippet and an API key without a sales gate. No reference captured here — directional, flagged so it isn't asserted as fact.
