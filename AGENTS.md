# Lazyweb skill pack — agent guidance

## Lazyweb report principles (governs any agent building a Lazyweb report)

Every Lazyweb skill produces an HTML report. Whatever the skill, the report must embody these four principles — they are the product, not a style preference.

1. **Show, don't tell.** Prove every claim with a real visual reference or data, never prose alone. Each assertion (pattern, anti-pattern, idea, hypothesis, "what's working", convention check, recommendation, or A/B learning) carries the screenshot(s)/experiment that demonstrate it, sitting *with* the claim. Quantify prevalence ("5 of 9 references") instead of asserting it ("near-universal"). Never render a proposed layout as ASCII/box-drawing `<pre>` art — use an HTML/CSS prototype, mock-frame, or generated image.

2. **Be opinionated; carry the decision.** Take a clear ranked stance and lead with a single recommended path, marked as such in the human-visible body (not only in the machine handoff block). Tell the user what to do first and what to skip, with a one-line reason each. Never hand over an undifferentiated menu of co-equal options and make the reader choose.

3. **Maximize confidence with evidence.** Back each recommendation with what worked for OTHER apps (real screenshots) PLUS supporting data — prevalence across the corpus, and A/B experiment learnings for growth/monetization screens. When experiment data is unavailable, say so and fall back to a stated prevalence count; never ship a recommendation with no visual and no number behind it.

4. **Be truth-seeking.** Never overclaim. Label evidence strength honestly (measured vs directional vs single-source/off-category) on each claim, and flag a weak, thin, single-source, or context-mismatched corpus up front. Tag any brand inferred from a URL/vision description as unverified. Never fabricate a reference, a metric, or a company name. The machine-readable handoff and the human-facing body must agree about confidence.

Shared report furniture (use the existing tokens `--ink:#1f2328; --mut:#57606a; --line:#d0d7de; --soft:#eef4fb; --accent:#0969da`): most Lazyweb skills continue to use the shared `.deck`, `.legend` + `.rec`, `.ebadge`/`.corpus`, `.flip`, and `.mock` components defined in their own `SKILL.md` files.

**Design-research v2 exception:** `lazyweb-design-research` now uses a prototype-first report instead of the older busy furniture. Its visible report sections are `Goal`, `Recommendation`, optional `Inspo`, and `Interesting Patterns`. When a current page/screenshot exists, `Recommendation` shows a stacked `Control` then prototype options; do not place them side by side. For broad design-language work, it must reason through 2-4 distinct hypotheses, choose a default recommendation, generate a detailed agent prompt for each viable direction, and render 2-4 `.prototype-option` outputs for the user to choose from. Use generated bitmap prototype images via `.prototype-img` when image generation is available; use HTML/CSS `.prototype` only as a fallback or when implementation-ready code is explicitly requested. Multiple prototype options must sit in a horizontal snap scroller where the recommended option is first and takes most of the viewport. Generate prototype images in parallel at medium effort by default, low effort when the user asks for speed, and high effort only for explicitly final/production-grade design. If native imagegen is unavailable, try Codex CLI with `model_reasoning_effort="medium"`, then configured Nano Banana/Gemini APIs, then prompt-ready fallback cards. Every image prompt must include `CONTROL`, `WHAT TO IMPROVE`, `HOW TO IMPROVE`, `INSPIRATION`, and `ATTACHMENTS`, with attachments labeled `CONTROL`, `INSPO A`, and `INSPO B` when uploads are supported. Prototype options must differ by conversion mechanism and information architecture, not only color, typography, or theme. It avoids side text columns, `Rec`/`Why` blocks, broad recommendation lists, recommendation metric chips such as `8 selected references`, and heavy evidence containers. Its inspo map uses image-only `.inspo-point` / `.inspo-img` references, with no visible company labels inside the map, and deliberately crops long desktop/web screenshots to the decisive viewport instead of shrinking full long-scroll pages. Its Interesting Patterns section uses larger cropped reference images through `.pattern-shot` / `.pattern-crop`; it must not use the uploaded/current/control picture there, must not draw red boxes or numbered callouts, and must not place explanatory copy under the crop. It uses `.hypothesis-strip`, `.hypothesis`, `.prototype-options`, `.prototype-option`, `.prototype-img`, `.build-prompt`, `.prototype`, `.inspo-map`, `.inspo-point`, `.inspo-img`, `.pattern-shot`, and `.pattern-crop` components instead of mandatory `.deck`/`.legend`/`.rec` cards. When the corpus is not thin, it should select a broader proof set (roughly 6-10 references, with 4-8 high-signal pattern crops) so the report does not feel under-evidenced. Do not reintroduce a visible standalone Agent Instructions block, Key Examples section, Findings table, Sources section, broad recommendation list, or long prose analysis into this skill.

---

These principles live in each skill's report contract. When editing a skill, keep its local component rules internally consistent. For `lazyweb-design-research`, keep the v2 prototype/inspo/pattern-crop components in sync with this file; for the other skills, keep their existing shared evidence components in sync.
